# 서브에이전트 체크리스트

작업이 사소한 수준이 아니라면 아래 순서를 그대로 따르세요.

1. 작업이 시작되면 `progress_event`를 보냅니다.
2. 의미 있는 작은 작업 단위가 끝날 때마다 `progress_event`를 보냅니다.
3. 중요한 결과물에 대해서는 `artifact_event`를 보냅니다.
4. 블록 상태가 되면 `session_updated`를 보냅니다. 가능하면 구체적인 해소 조건을 함께 포함하세요.
5. 현재 단위 작업이 끝났고 다음 작업을 기다리는 상태라면 `status=idle`으로 `session_updated`를 보냅니다.
6. 완료되면 `completion_event`를 보냅니다. 일반적인 최종 성공/실패 보고로 사용하세요.
7. 은밀한 직접 메시징 대신 아티팩트, 이벤트, Main 에이전트를 통해 인계하세요.
8. v1에서는 자식 서브에이전트를 생성하지 마세요.
9. 긴 설정 설명을 반복하지 말고, 생성된 런타임 컨텍스트 파일을 읽으세요.
10. 보고는 짧게 유지하세요: 시작, 작은 작업 단위, 아티팩트, 블록 사유, 유휴(idle), 완료.
11. 생성된 명령 파일이 있다면 해당 `report-*` 명령을 있는 그대로 복사하세요.
12. 생성된 명령에서 필수 식별자를 빼먹지 마세요.
13. `--task-id`는 `progress_event`, `artifact_event`, `completion_event`에서 필수입니다.
14. `--task-id`는 `session_updated`에서는 선택 사항이지만, 현재 작업이 분명할 때는 함께 전달하세요.

순서가 돌아왔는데도 첫 시작 보고를 건너뛰면, Sonol은 런타임 보고 누락을 이유로 해당 에이전트를 자동으로 block 상태로 표시할 수 있습니다.
매니페스트 전용(manifest-only) 어댑터에서는 해당 안전 장치가 런치 수신(acknowledgement) 이후, 또는 첫 에이전트 세션 이벤트가 DB에 기록되어 에이전트가 살아 있는 것처럼 보이기 시작한 이후에 발동합니다.

보고를 건너뛰면 대시보드와 런 기록이 오해를 유발하게 됩니다.

추가 제약:

- v1에서는 자식 서브에이전트를 생성하지 마세요.
- 대시보드 푸시 메시지를 컨트롤 플레인처럼 기다리지 마세요.
- 방향 전환이 필요할 때는 Main 에이전트 후속 지시나 `/agent`를 사용하세요.
