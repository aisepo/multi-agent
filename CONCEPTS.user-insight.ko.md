# 사용자 행동 분석 파이프라인 — 아이디어 추출 문서

이 문서는 `CONCEPTS.ko.md` 의 멀티 에이전트 구조를 **운영 중인 웹 서비스의 사용 패턴 감지·개선점 파악**에 적용할 때의 설계 개념만 정리합니다. 코드·도구 이름·스키마는 포함하지 않습니다.

## 0. 해결하려는 문제

> "서비스를 출시한 뒤, 사용자들이 실제로 어떻게 쓰는지를 감지해 개선 후보를 찾아낸다. 단, 에이전트의 발견이 그대로 제품을 흔들지 않게 한다."

핵심 난점은 "발견의 품질"보다 **발견과 결정 사이의 경계**입니다. LLM 에이전트는 그럴듯한 가설을 쉽게 만들지만, 그 가설을 자동으로 A/B·UI 변경에 연결하면 제품이 에이전트 환각에 흔들립니다. 이 문서의 모든 판단은 **조언자로서의 에이전트**에 관한 것입니다.

---

## 1. 에이전트 구성 — 네 개의 역할

단일 "분석 에이전트" 대신, 책임을 네 개로 쪼갭니다. 이유는 역할 간 **의견 불일치가 가시화**되어야 사람이 개입 지점을 판단할 수 있기 때문입니다.

1. **관찰 에이전트 (Observer)** — 로그·분석 이벤트·세션 녹화 요약을 읽고, 이상 패턴·퍼널 이탈·반복 행동·체류 편차를 **서술적으로** 요약. 판단하지 않음.
2. **가설 에이전트 (Hypothesizer)** — 관찰 요약을 받아 "왜 이 지점에서 이탈하는가"에 대한 복수 가설을 생성. **의도적으로 N개** (예: 3~5개) — 하나만 쓰면 사람이 검토 없이 채택함.
3. **검증 에이전트 (Validator)** — 가설을 **원본 데이터로 대조**. 근거가 약한 가설은 "증거 불충분"으로 기각. 이 단계는 LLM만으로 하지 않고 **수치 쿼리·통계 테스트**를 강제.
4. **통합자 (Synthesizer, 현 세션=메인)** — 살아남은 가설을 **제품 로드맵 제안** 형태로 통합. 여기서 끝. 이후 결정은 사람.

`CONCEPTS.ko.md` 의 "메인은 암묵적" 원칙이 그대로 적용됩니다 — 에이전트 목록에 "통합자"를 명시적으로 넣지 않습니다.

---

## 2. 조언 계층만 존재한다 (반성 원칙의 재적용)

`CONCEPTS.ko.md §11` 의 핵심: **반성은 조언이지 법이 아니다**. 같은 원칙을 사용자 분석에 그대로 적용합니다.

- 에이전트의 발견은 **힌트**로만 존재한다. 제품 코드·UI·실험 설정을 자동으로 바꾸지 않는다.
- "자동 A/B 시작", "문구 자동 교체" 같은 기능은 **만들지 않는다**. 만드는 순간 자기증폭 루프가 생긴다.
- 로드맵 반영은 **사람의 승격(promotion)** 을 거친다. 승격된 항목만 제품 백로그로 이동하고, 그때서야 변경 권위를 얻는다.
- 에이전트는 사후 설명도 한다 — "내가 왜 이 가설을 버렸는가" 도 같이 기록. 설명이 나중 반성의 재료가 된다.

---

## 3. 데이터 경계와 프라이버시

개인정보·사용자 원문은 LLM 경계를 **넘지 않게** 설계합니다. 이는 선택이 아니라 법·계약·신뢰의 문제입니다.

**세 단계 정제**

1. **수집층**: 원본 이벤트 (URL, 체류 시간, 클릭 시퀀스, 에러) — 내부 저장소에만.
2. **집계층**: 세션·코호트·기능별로 익명 집계 — PII 제거, 재식별 불가능 수준까지.
3. **요약층**: 집계를 자연어 요약 — **LLM에 전달되는 유일한 형태**.

관찰 에이전트는 3단계만 받습니다. 가설·검증 에이전트는 집계층까지 접근할 수 있지만 프로그램 호출로만 — LLM 프롬프트에는 여전히 요약만 넣습니다.

**적어선 안 되는 것**: 사용자 입력 원문(검색어, 댓글, 메시지), 이메일, IP, 기기 식별자, 결제 정보. 이들이 필요하면 별도 보안 파이프라인을 만들고 에이전트와 분리합니다.

---

## 4. 실행 주기 — 대부분 배치가 정답

실시간 대시보드형 분석은 매혹적이지만 비용·노이즈 양쪽에서 비현실적입니다.

- **일 1회 / 주 1회 배치** — 대부분의 인사이트가 "하루 지나도 유효". 비용이 예측 가능하고 LLM 호출을 모아 할인·캐싱을 쓸 수 있음.
- **이벤트 트리거형** — "결제 이탈률이 임계치 초과", "에러 스파이크" 같은 **숫자 조건**이 먼저 켜지고, 그때만 에이전트가 깨어남. LLM은 감시자가 아니라 **설명자**.
- **실시간 파이프라인** — 고위험·고비용. 운영 한 번 해 본 뒤에만 고려.

경계 원칙: **숫자가 먼저 울리고, 에이전트가 나중에 설명한다.** 에이전트가 숫자를 계속 쳐다보게 두지 않습니다.

---

## 5. 측정 피드백 루프를 끊는다

에이전트의 위험한 능력 중 하나는 "이 지표를 새로 계측해야 한다"는 제안입니다. 이게 자동으로 반영되면:

- 에이전트가 **자신이 보고 싶은 지표**를 만들어 냄
- 다음 주기에 그 지표를 보고 같은 결론을 강화
- 실제 사용자와 괴리된 자기완결적 세계관이 형성됨

**끊는 법**: 계측 변경은 **반드시 사람 승인**. 에이전트는 "이 지표가 있으면 좋겠다"는 제안만 하고, 실제 계측 코드는 엔지니어가 추가. 추가된 지표의 결과를 같은 에이전트가 다시 해석하는 것까지는 허용하되, 해석 품질에 대한 **외부 검토**(다른 에이전트 또는 사람)를 강제.

---

## 6. 이벤트 계약 — 세 가지 종

운영 중 서비스에서는 `CONCEPTS.ko.md §9` 의 4종(progress/artifact/completion/session)과 별도로, **분석 측 이벤트** 3종을 둡니다.

1. **관찰 요약 (observation_summary)** — "이번 주기에 이런 패턴이 있었다". 판단 없음.
2. **가설 기록 (hypothesis)** — 가설 본문·근거·증거 강도·검증 상태. 상태는 `제안됨 / 증거 있음 / 증거 없음 / 승격 / 폐기`.
3. **인사이트 카드 (insight_card)** — 통합자가 승격 후보로 올린 최종 형태. 사람이 **버튼 하나로 백로그에 보냄**.

각 이벤트에 스키마 버전을 박아 둡니다 — 분석 규칙이 자주 바뀌므로 과거 기록의 해석 가능성이 중요합니다.

---

## 7. 대시보드 — "이번 주의 발견" 패널

운영자(대부분 본인) 관점에서 한 화면에 다음을 배치합니다.

- **숫자 알람 구역** — 임계 초과 지표만. 에이전트의 의견 없음.
- **이번 주기 관찰** — 관찰 에이전트 요약 카드들.
- **가설 보드** — 상태별로 칼럼(제안됨 / 증거 있음 / 증거 없음 / 폐기). 카드에 **근거 링크** 필수.
- **승격 대기** — 통합자가 올린 인사이트 카드. "백로그로 보내기 / 보류 / 폐기" 버튼.
- **반성 기록** — 과거에 승격했다가 **실패한** 인사이트. 이걸 같은 자리에 두는 게 핵심 — 에이전트와 본인 모두의 맹점을 보여 줌.

대시보드는 `CONCEPTS.ko.md §12` 원칙대로 **권위가 없음**. 승격 버튼이 실제로 하는 일은 별도 백로그 시스템에 항목을 보내는 것 뿐이고, 분석 DB는 그 결정을 기록할 뿐 제품을 바꾸지 않습니다.

---

## 8. 주요 트레이드오프

| 선택 | 장점 | 단점 |
|---|---|---|
| 실시간 vs 배치 | 빠른 감지 | 비용 폭증, LLM 환각이 즉각 로드맵 오염 |
| LLM이 원본 이벤트 vs 집계만 | 맥락 깊음 | 프라이버시·비용·재현성 악화 |
| 가설 1개 vs N개 | 단순 | 검토 없이 채택됨 |
| 통합자가 한 명 vs 다수 투표 | 빠름 | 편향 고착 |
| 자동 승격 vs 사람 승격 | 빠른 반영 | 자기증폭 오류 |
| 전용 분석 에이전트 vs 범용 LLM | 전문성 | 유지보수 대상 증가 |

**기본 추천**: 배치, 집계만, 가설 3~5개, 통합자 1명, 사람 승격, 초기엔 범용 LLM + 좋은 프롬프트.

## 9. 단계적 도입 순서

한 번에 전부 만들지 않습니다. 사업 초기에 적합한 순서:

1. **계측부터**. 이벤트 로깅·퍼널 추적·에러 수집을 먼저 제대로. 에이전트 없이도 숫자만 보면 80%는 나옴.
2. **숫자 알람**. 임계 기반 룰. 여전히 에이전트 없음.
3. **주 1회 배치 관찰 에이전트**만 추가. 요약만. 가설·검증 없음.
4. 요약이 쓸만하다 싶으면 **가설·검증** 추가.
5. 승격·반성 루프 추가.
6. 필요하면 실시간·전용 모델 고려.

각 단계에서 "이게 없으면 안 되는가"를 먼저 물어봅니다. 웹 서비스 초기엔 대개 1·2단계로 충분합니다.

## 11. 하지 말 것

- 에이전트가 제품 코드·설정·피처 플래그·실험 스크립트를 **직접 바꾸게** 한다. (자기증폭 위험)
- 원본 사용자 입력·PII를 LLM 프롬프트에 넣는다. (법·신뢰)
- 에이전트 발견을 검토 없이 **자동으로 팀 채널에 브로드캐스트** 한다. (노이즈 폭증)
- 한 에이전트가 관찰·가설·검증을 모두 한다. (자기 확증)
- 대시보드에 **성공한 인사이트만** 보여 준다. 실패·폐기 기록이 없으면 맹점 교정이 불가능.
- "이 지표가 유의미해 보인다"는 LLM 의견만으로 계측을 늘린다.

---

## 맺음말

사용자 행동 분석에 에이전트를 쓰는 건 좋은 선택이지만, 제품을 바꾸는 권위까지 넘겨주는 건 나쁜 선택입니다. 이 문서의 모든 판단은 한 줄로 요약됩니다 — **에이전트는 보고 설명하고 제안할 수 있다. 결정은 언제나 사람.**
