# 멀티 에이전트 오케스트레이션 — 아이디어 추출 문서

이 문서는 본 레포의 **소스 코드·표현·식별자를 차용하지 않고**, 아키텍처 수준의 개념과 설계 판단만을 정리합니다. 새 프로젝트를 백지 상태에서 설계할 때 참고용으로 쓰는 "개념 시방서"에 해당합니다. 이름·스키마·파일명·변수명은 모두 일반 용어로 치환되어 있습니다.

## 0. 해결하려는 문제

> "하나의 사용자 요청을 복수 에이전트로 쪼개 동시 실행하고, 진행 상태를 관찰·승인·수정·재시도할 수 있으며, 결과를 안전하게 하나의 산출물로 통합한다."

이 문제의 핵심 난점은 기술보다 **권한(authority)·상태(state)·신뢰(trust) 경계**에 있습니다. 아래 개념들은 대부분 그 경계를 명확히 하려는 판단입니다.

---

## 1. 두 개의 역할을 물리적으로 분리한다

한 덩어리의 시스템을 **오케스트레이터**와 **서브 러닝 리포터** 두 모듈로 쪼갭니다.

- **오케스트레이터**: "누가 무엇을 할지"를 결정·승인·위임·통합
- **서브 리포터**: 위임받은 서브 에이전트가 "내가 무엇을 했는지"만 보고

왜 분리하나 — 한쪽이 다른 쪽을 의존하지 못하게 해서 순환 권한과 무한 재귀 위임을 막습니다. 서브 에이전트는 "기획"을 할 수 없고, 오케스트레이터는 "실행 리포트 스트림"에 직접 쓰지 않습니다. 이 단방향성이 디버깅·감사·테스트를 근본적으로 쉽게 만듭니다.

---

## 2. 권한을 네 계층으로 쪼갠다

한 시스템 안에서 "누가 권위를 갖는가"를 컴포넌트별로 명시적으로 나눕니다.

| 계층 | 책임 | 위치 |
|---|---|---|
| 창의 (Creative) | 초기 요청 해석, 에이전트 구성 초안 | 로컬 (현재 세션 LLM) |
| 정규화 (Normalization) | 초안 검증·바인딩·계약 확정 | 선택적으로 원격 서비스 |
| 실행 (Execution) | 실제 서브 에이전트 기동·진행·완료 | 로컬 |
| 상태 (State) | 계획·런·이벤트의 권위적 저장 | 로컬 |

핵심 원칙: **창의는 정규화를 대신할 수 없고, 정규화는 창의를 대신할 수 없다.** 정규화 서비스가 초안까지 작성하면 의도가 사라지고, 창의 측이 자체 검증까지 하면 일관성이 깨집니다. 둘은 서로의 역할을 침범하지 않습니다.

그리고 **상태는 반드시 로컬에서 권위를 가진다**. 원격 서비스는 stateless한 "순수 함수" 역할로 두고, 장애가 나도 로컬만 살아 있으면 시스템이 동작합니다.

---

## 3. 세 가지 데이터 형태를 분리한다

같은 작업을 서로 다른 세 개의 자료 구조로 표현합니다. 각 형태는 **다른 생애주기**를 갖습니다.

1. **창의 초안 (Creative Draft)** — 사람/LLM이 자연스럽게 표현한 형태. 어휘가 느슨하고 일부 필드만 채워도 됨. *편집 가능, 지속성 없음.*
2. **정규화된 계획 (Normalized Plan)** — 실행 엔진이 이해하는 엄격한 형태. 모든 필드가 강타입·강검증. *편집 가능, 승인 전까지.*
3. **불변 런 스냅샷 (Immutable Run Snapshot)** — 실행 순간 고정된 복사본. 한 번 만들어지면 수정되지 않음. *읽기 전용, 감사 가능.*

왜 세 개인가 — 초안 단계에서 편집을 막으면 사용자가 답답하고, 런 단계에서 편집을 허용하면 실행 결과가 어느 계획에 대한 것인지 추적 불가능해집니다. **"언제 얼어붙는가"** 를 데이터 모델 수준에서 강제합니다.

계획의 어휘와 초안의 어휘는 **의도적으로 달라야** 합니다. 같으면 사용자가 정규화 전후의 차이를 인지하지 못합니다.

---

## 4. 이중 승인 게이트

승인은 한 번이 아니라 **두 장소에서** 받습니다.

1. **GUI(대시보드) 승인**: 사용자가 구조·역할·의존 관계를 눈으로 보고 수정·승인
2. **CLI(터미널) 확정**: 사용자가 명시적 키워드(예: "승인")를 입력해 런 생성을 트리거

왜 이중인가 — GUI만 있으면 실수 클릭으로 런이 시작되고, CLI만 있으면 구조 파악이 어렵습니다. **"본 적 있다"와 "지금 실행하라"를 분리**합니다. 두 게이트를 서로 다른 채널(HTTP vs 터미널)에서 받는다는 점이 중요합니다 — 한 채널이 탈취되어도 런은 시작되지 않습니다.

---

## 5. 워크스페이스 바인딩

여러 프로젝트에서 동시에 시스템을 쓸 때를 위해, **작업 디렉터리마다 격리된 인스턴스**를 만듭니다. 바인딩이란 다음 세 경로를 한 묶음으로 해석해 주는 해석기입니다.

```
workspace_root  →  state_db_path  →  runtime_root
                                  →  dashboard_port (stable, derived)
```

포인트는 포트·DB 경로를 **작업 디렉터리 해시에서 결정론적으로 유도**한다는 것입니다. 그러면 사용자는 포트를 외울 필요가 없고, 다른 프로젝트와 충돌하지 않으며, 재시작해도 항상 같은 URL을 얻습니다.

해시 기반 포트 유도는 작지만 큰 UX 개선입니다. "대시보드 열기"가 항상 같은 링크를 내놓습니다.

---

## 6. 어댑터 패턴 — 프로바이더 중립성

서로 다른 LLM 프로바이더(예: Codex, Claude Code, 기타)를 **공통 실행 계약**으로 감쌉니다.

- 어댑터는 정규화된 계획을 받아 **프로바이더별 런치 패킷**으로 변환
- 모든 어댑터는 같은 이벤트 스키마를 써서 결과를 보고
- 대시보드·상태 DB는 어댑터 종류를 모름

**핵심 변형: "매니페스트 전용(manifest-only)" 디스패치.** 어댑터가 서브 에이전트를 직접 스폰하지 않고, **런치 레시피만 반환**하는 모드를 둡니다. 실제 위임은 상위 세션에서 수동으로 일어납니다. 이는 다음 두 조건에서 필요합니다.

1. 프로바이더가 자식 프로세스를 허용하지 않는 CLI/IDE 환경
2. 사용자가 각 서브 에이전트 기동을 개별적으로 확인하고 싶을 때

"자동 실행 가능"은 옵션이지 기본이 아니라는 판단입니다.

---

## 7. 런타임 생성 프롬프트 — 계획과 실행의 분리

계획에 담긴 프롬프트 문자열은 **초안이자 참고용**일 뿐입니다. 런이 생성되는 순간 **별도 파일**로 프롬프트가 물질화(materialize)되고, 이후 위임에서는 이 파일만 권위를 갖습니다.

```
<runtime_root>/<run_id>/<agent_id>.txt   ← 권위
<plan>.<agent>.prompt                     ← 편집 가능하나 실행에 영향 없음
```

왜 — 계획 편집이 런 진행 중인 서브 에이전트의 지시문을 바꾸면 결과가 재현 불가능해집니다. **"계획은 살아 있고 런은 얼어 있다"** 는 원칙을 파일 시스템 수준에서 강제합니다.

---

## 8. 이벤트 계약 — 보고의 4가지 종

서브 에이전트는 딱 네 종류의 이벤트만 올립니다. 스키마는 버전을 명시합니다.

1. **진행(progress)** — "지금 몇 단계 중 몇 번째"
2. **산출물(artifact)** — "어떤 파일/결과를 만들었다"
3. **완료(completion)** — "성공 또는 실패로 끝났다"
4. **세션(session)** — "막혔다 / 놀고 있다" (정상 완료는 여기 쓰지 않음 — completion을 쓴다)

왜 종류를 최소화하나 — 서브 에이전트가 보고해야 할 상태의 **축(axis)** 을 줄이면, 대시보드 UI도 단순해지고, 통합 로직도 하나의 이벤트 테이블을 스캔하면 됩니다.

스키마 버전을 `event.schema_version` 로 박아 두면, 추후 마이그레이션 시 기존 이벤트를 그대로 보존한 채 새 이벤트만 새 버전으로 쓸 수 있습니다.

---

## 9. 메인 에이전트는 암묵적이다

서브 에이전트 목록에 "메인"을 **절대 넣지 않습니다**. 메인은 항상 현재 세션 자신이고, 최종 통합을 책임집니다.

왜 — 메인을 명시적으로 쓰게 하면 사용자(혹은 LLM)가 "메인"의 이름/역할을 고민하게 되고, 결과적으로 시스템이 자기 자신을 하나의 노드로 취급하게 됩니다. 이는 위임 그래프를 순환적으로 만듭니다. **"나는 위임자이지 피위임자가 아니다"** 를 데이터 모델로 강제합니다.

---

## 10. 실행 제어가 아닌 "보고 계약"

서브 에이전트를 낮은 수준에서 제어(스레드 제어, 출력 후킹, 최종 응답 JSON 스크래핑)하지 않습니다. 대신 **명시적 보고 스크립트**를 호출하게 합니다.

- 후킹은 프로바이더 버전에 취약함
- 최종 답변 스크래핑은 파싱 실패 시 이벤트가 사라짐
- 명시적 호출은 실패 시 명시적 오류로 나타남

프로바이더가 스크립트 호출조차 막힌 경우를 위해 **JSON 표준 입력(stdin) 폴백**을 둡니다. 채널은 다양해도 **데이터 포맷은 하나**입니다.

---

## 11. 반성(Reflection) 계층 — 조언으로만

같은 실수를 반복하지 않으려면 시스템이 과거 런을 돌아보는 장치가 필요합니다. 단, **반성은 학습이지 법이 아닙니다** — 권위 계층을 직접 수정하게 하면 잘못된 반성이 새 제약으로 굳어 **자기 증폭 오류(self-reinforcing bias)** 를 만들고, "런 스냅샷은 불변"이라는 원칙을 깨뜨립니다.

**배치 원칙**

- 반성은 **조언(advisory) 계층**으로만 둔다. 창의 계층이 다음 드래프트를 작성할 때 **힌트로 읽을 뿐**, 강제 제약이 되지 않는다.
- 실행·상태·정규화 계층은 반성을 **보지 않는다**. 이들은 오직 현재 런의 계약만 따른다.
- 반성 기록은 이벤트 스트림의 **곁가지**로 추가한다. 대표적인 종이 **경계 위반 기록(boundary_violation)** — 예: 창의가 정규화 어휘를 썼다, 서브가 기획을 시도했다, 승인 없이 런이 생성되었다.
- 반성이 힌트가 되려면 **승인·무효화·만료** 를 다룰 주체가 있어야 한다. 자동 생성된 반성을 사람이 읽고 "다음에 적용" / "무시" / "영원히 금지" 중 택하게 한다.

**반성의 수명**

반성은 프로젝트·주제·런 유형에 **스코프를 달아** 저장한다. "모든 런에 영원히 적용되는 반성"은 사실상 새로운 규칙이므로, 그건 반성이 아니라 **정규화 계약의 변경**으로 승격시켜야 한다 — 같은 계층에 섞지 않는다.

**구체적 형태 (예시)**

- 완료 이벤트 옆에 `boundary_violation` 종 추가, 단 권위는 없음
- 창의 계층 프롬프트에 "최근 N개 런의 위반 요약"을 주입
- 대시보드에 "이번 런에서 배운 것" 패널 — 사람이 승격/폐기 버튼으로 처분
- 승격된 반성만 정규화 계약으로 이동하고, 그때서야 강제력을 얻음

**요약**: 반성은 창의를 더 똑똑하게 만드는 **조언자**, 권위를 흔드는 **입법자**가 아닙니다.

---

## 12. 로컬 브리지 + SPA 대시보드

대시보드는 전형적인 "로컬 서버 + 정적 SPA" 구조입니다.

- 로컬 프로세스가 HTTP + WebSocket을 호스팅
- React(또는 선호 프레임워크) SPA가 브리지에 붙음
- SPA 자체는 **권위가 없음** — 새로고침하면 상태가 DB에서 다시 로드됨
- 브라우저 저장소는 임시 UI 상태만 보관

원격 SaaS 대시보드를 쓰지 않는 이유 — **코드·파일 경로·런 산출물이 로컬에 있기 때문**. 인증·업로드·왕복 지연을 감수할 만한 이득이 없습니다. 로컬 브리지가 표준 답입니다.

---

## 13. 작은 운영 유틸리티들

큰 아키텍처와 별개로, 다음 소도구들이 시스템을 실용적으로 만듭니다.

- **동시성 락 파일** — 원격 정규화 등 배타 작업에 stale-timeout 달린 락. 프로세스가 죽어도 n분 후 자동 회수.
- **플랫폼 경로 정규화 계층** — WSL `/mnt/c/…` ↔ Windows `C:\…` 양방향 변환. 보존해야 할 것은 "원본 형태", 비교용은 "정규화된 형태"로 분리.
- **공개 릴리스 필터** — 레포에서 내부 전용 파일(인증 키, 서버 구현)을 자동으로 제외하는 빌드 스크립트.
- **진단 커맨드** — 현재 바인딩, 런 매니페스트, 최근 이벤트를 한 번에 볼 수 있는 CLI. "어디서 뭐가 잘못됐나"를 사람 눈으로 빠르게 잡기 위함.

---

## 14. 차별화하기 좋은 포인트

위 개념을 그대로 따라하면 복제에 가까워집니다. 자기 것으로 만들려면 아래 중 **하나 이상**을 본인만의 선택으로 바꾸는 걸 추천합니다.

- **스택 교체** — Node → Python(FastAPI/SQLModel) 또는 Go(+SQLite)로 구현. 대시보드도 Vue/Svelte/HTMX.
- **상태 저장소 교체** — SQLite 대신 이벤트 스토어(예: SQLite+append-only log, LiteFS, Redpanda)로 확장해 시간여행 감사.
- **정규화 계층의 위치** — 원격 서비스 대신 로컬 in-process 함수로 둘지, 여러 LLM 중 투표를 하게 할지.
- **승인 게이트 수** — 1단계 / 2단계 / 3단계(대시보드·CLI·사내 채팅봇)로 다르게.
- **이벤트 종류 확장** — "checkpoint", "rollback", "human-input-required" 같은 축을 추가.
- **어댑터 범위** — 로컬 CLI 외에 컨테이너, 원격 VM, 브라우저 확장 등으로 확장.
- **메인 에이전트 모델** — 암묵이 아닌 "감독자 에이전트"를 명시적으로 두는 변형.
- **반성의 강도** — 조언만 / 조언+승격 경로 / 자동 제약(권장 안 함) 중 어디까지.
