# 대시보드 워크플로

운영자용 대시보드는 기존 로컬 빌드와 동일한 전체 대시보드 표면을 유지하되, 원격에 호스팅되며 워크스페이스 범위의 로컬 루프백 브리지와 통신합니다. 플랜, 승인, 런, 런타임 이벤트의 권위는 로컬 SQLite 데이터베이스에 계속 남습니다.

## 탭 구성

- `Plan`: 요청 요약, 단일/멀티 추천, 추천 에이전트
- `Agents`: 역할 목록, 프로바이더 에이전트 타입, 모델 선택, 리즈닝 강도, 샌드박스 모드, MCP/스킬 표면, 의존성 뷰, 경로 정책
- `Run`: 런치 모드, 런 생성, 상태 변경, 재시도 계보
- `Monitor`: 실시간 런 상태, 현재 작업, 최근 이벤트, 블록 사유
- `Outputs`: 아티팩트 및 완료 요약
- `Advanced Controls`: DB 경로, 런타임 바인딩, 실행 관련 심화 가정

## 운영자 규칙

- 자동 저장(auto-save)은 드래프트 플랜만 변경합니다.
- 원격 대시보드 URL은 루프백 브리지 부트스트랩 프래그먼트를 통한 워크스페이스 범위의 링크로 취급해야 하며, 모든 워크스페이스가 공유하는 하나의 고정 포트로 간주해서는 안 됩니다.
- 브리지 주소와 토큰은 원격 URL의 프래그먼트(fragment)에 담아, 브라우저가 이를 HTTP 요청 라인이나 리퍼러(referrer)를 통해 원격 서버로 전송하지 않고 읽을 수 있도록 해야 합니다.
- `#bridge=...&bridge_token=...`을 포함한 원본 리뷰 URL이 현재 세션의 정식 부트스트랩 링크입니다. 같은 컴퓨터에서 새 탭이나 브라우저로 대시보드를 다시 열어야 한다면 이 URL을 그대로 유지하세요.
- 대시보드 탭이 이미 열려 있다면, `#bridge=...` 프래그먼트가 포함된 원본 리뷰 URL을 그 탭에 다시 붙여 넣어 현재 링크의 브리지 부트스트랩을 재초기화할 수 있습니다. 프래그먼트가 없는 단순한 대시보드 경로만으로는 부족합니다.
- 부트스트랩 프래그먼트 없이 단순 대시보드 경로만으로는 그 자체로 공유 가능한 런치 링크가 될 수 없습니다.
- 리뷰 URL은 같은 컴퓨터 안에서 쓰이는 세션 링크입니다. 로컬 루프백 브리지와 현재 Sonol 터미널 세션에 의존하므로, 컴퓨터 간을 넘는 영구 링크로 취급해서는 안 됩니다.
- 대시보드 헬스 및 스냅샷 표면은 `dashboard_url`, `authoritative_db_path`, `workspace_root`를 노출합니다. 브라우저와 터미널이 동일한 워크스페이스 상태를 바라보고 있는지 이 필드들로 확인하세요.
- Sonol은 워크스페이스 로컬 권위 메타데이터를 `<workspace_root>/.sonol/authority.json`에, 런타임 권위 메타데이터를 `<runtime_root>/dashboard/authority.json`에 기록합니다. 운영자 CLI 진단은 휴리스틱 기반 DB 탐색으로 넘어가기 전에 먼저 이 권위 레코드를 사용해야 합니다.
- 브라우저는 정적 자산을 캐시할 수 있지만, 오케스트레이션 상태의 원본이 되어선 안 됩니다. 브리지 기반 SQLite 스냅샷이 권위를 가집니다.
- 대시보드는 플랜을 검토, 수정, 승인할 수 있지만, 터미널 확정(confirmation)을 대신해서는 안 됩니다.
- 대시보드 승인 이후, 대시보드는 현재 Sonol 터미널 세션에서 `승인`을 입력하도록 운영자에게 안내해야 합니다.
- 그 후 어시스턴트는 가장 최근의 활성 승인 플랜을 해석하고, 내부 터미널 확정 단계를 수행합니다.
- 터미널 확정은 런 스냅샷, 런타임 컨텍스트, 런치 매니페스트를 준비합니다. 이는 프로바이더 서브에이전트가 자동으로 생성되었음을 뜻하지 않습니다.
- 어댑터가 매니페스트 전용(manifest-only)이라면, 운영자에게 보이는 다음 단계는 "이미 작업이 시작됨"이 아니라 "지금 매니페스트에서 런치"입니다.
- 해당 모드에서는 자유 형식 프로바이더 위임이 유효하지 않습니다. 계획된 각 `agent_id`에 대해 런 범위 프롬프트 파일을 사용하세요.
- 터미널 확정 시에는 DB 바인딩을 해석할 때 터미널의 초기 DB 선택이 여전히 옳다고 가정하기보다, 가장 최근 승인 플랜의 `authoritative_db_path`를 우선해야 합니다.
- 터미널 확정은 즉시 런치하지 않고, 실행 가능한 안내와 함께 중단될 수 있습니다.
  예: 대시보드 승인이 없을 때, 초점이 맞지 않는 플랜일 때, 검증이 수정되어야 할 때, 또는 더 최신 편집 이후 운영자가 새로고침 후 재확정해야 할 때.
- 런치는 항상 런 스냅샷을 생성합니다.
- `<runtime_root>/<run_id>/prompts/...` 아래의 런 범위 프롬프트가 활성 런의 권위입니다.
- 런타임 지침은 `<runtime_root>/<run_id>/prompts/<agent_id>.txt` 아래의 런 범위 프롬프트에서 읽어야 합니다.
- 활성 런 상태는 DB 기반 이벤트 스트림에서 가져와야 합니다.
- 매니페스트 전용 어댑터에서는, `prepared` 런이 실제 서브에이전트 진행 보고가 오기 전이라도 대시보드에 런/세션 상태를 표시해야 합니다. "아직 서브에이전트 보고가 없음"은 "대시보드에 상태가 없음"과 다릅니다.
- 정식 SQLite 이벤트 테이블은 `events`입니다. 구버전 진단을 위해 `runtime_events`라는 호환 뷰가 존재할 수 있지만, 새 툴은 공식 API와 `events` 테이블 의미를 우선해야 합니다.
- 원격 대시보드 미러는 로컬 SQLite 브리지 없이는 권위를 가지지 못합니다.
- 대시보드는 현재 어댑터 제어 표면과 Sonol 로컬 거버넌스 필드 범위 내에서 지원되는 서브에이전트 제어를 수정할 수 있습니다.

## 대시보드 제어 경계

대시보드가 할 수 있는 일:

- 멀티 에이전트가 유익한지 표시
- 지원되는 서브에이전트 제어 표시 및 수정
- 현재 실효 동시성 및 깊이 정책 표시
- 운영자 확인용으로 현재 워크스페이스 루트, 대시보드 URL, 권위 DB 바인딩 표시
- 블록 사유, 아티팩트, 완료 요약 표시
- 터미널에서 무엇을 확정할지 운영자에게 안내
- 현재 어댑터가 원격 취소를 노출하지 않을 때에만 로컬 중지 컨트롤 표시
- 브리지 토큰은 원격 UI가 로컬 루프백 브리지에 인증하기 위한 용도로만 사용하고, 런타임이나 플래너 자격 증명으로 재사용하지 않음

대시보드가 해서는 안 되는 일:

- 기존 터미널 세션에 텍스트 주입
- 어댑터가 명시적으로 지원한다고 말하지 않는 한, 프로바이더 서브에이전트 스레드를 직접 조종하거나 종료
- 로컬 중지 버튼이 원격 프로바이더 스레드 취소를 의미한다고 시사
- 권고 수준의 경로 정책을 네이티브 강제 실행처럼 주장
- 로컬 SQLite 저장소를 브라우저 저장소나 원격 영속화로 대체

## 대체 패턴

- 대시보드 승인 + 명시적 터미널 확정을 함께 사용
- 직접적인 에이전트 제어 대신 대시보드 가시성 + 런타임 이벤트 사용
- DB/워크스페이스 식별이 의심되면 먼저 `show-authority.mjs` 사용
- 디스크의 `launch-manifest.json`이 필요하다고 가정하기 전에 `show-run-launch-manifest.mjs` 또는 `/api/runs/<run_id>/launch-manifest` 사용
- 서브에이전트의 방향을 다시 잡아야 할 때는 `/agent` 또는 일반적인 메인 스레드 후속 지시를 사용
- 대시보드는 기존 터미널 세션에 텍스트를 주입할 수 있다고 가정해서는 안 됨
- 대시보드는 지원되는 서브에이전트 제어를 표시할 수 있지만, 지원되지 않는 제어는 묵시적으로 두지 말고 "사용할 수 없음"으로 명시해야 함
