# 에이전트 구성 (Agent Shaping)

요청을 단일 에이전트로 유지할지, 여러 트랙으로 나눌지 결정할 때 이 가이드를 사용하세요.

## 공개 지침

- Main 에이전트는 언제나 최종 통합 역할을 맡습니다.
- 작업이 명확하게 독립적인 트랙으로 나뉠 때에만 서브에이전트를 추가하세요.
- 고정된 묶음(bundle)을 강제하기보다 업무에 맞는 역할 이름을 사용하세요.
- 운영자가 여전히 검토할 수 있을 정도로 전체 계획 규모를 작게 유지하세요.
- 계획을 너무 공격적으로 쪼개기 전에, 유사한 트랙은 먼저 병합하세요.
- 작업에 실제로 수정이나 회귀 위험이 있을 때에만 검증(verification) 또는 리뷰(review) 트랙을 추가하세요.
- 서브에이전트 깊이(depth)는 얕게 유지하세요.
- 쓰기(writer) 트랙이 충돌할 수 있는 상황에서는 동시성을 낮추세요.
- 로컬 DB와 런타임 보고를 진실의 원본(source of truth)으로 보존하세요.

## 일반적인 트랙 예시

- 계획(planning) 또는 순서 결정(sequencing)
- 조사(research) 또는 사실 확인(fact-finding)
- 구현(implementation)
- 검증(verification)
- 리뷰(review)
- 문서 작성(docs) 또는 인계(handoff)
- 운영(ops) 또는 환경 점검

## 참고 사항

- 정확한 개수, 제목, 책임은 현재 요청과 로컬 AI가 작성한 크리에이티브 드래프트에서 도출되어야 합니다.
- 호스팅 플래너는 해당 드래프트를 정규화(normalize)하고 바인딩(bind)하는 역할만 수행해야 하며, 원격 측에서 새로 작성하지 않아야 합니다.
